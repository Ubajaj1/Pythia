Step 1 — Analyze  ·  1 LLM call
Scenario Analyzer
  • Reads plain English prompt
  • Extracts scenario type, title, description
  • Defines 5-point stance spectrum
  • Creates 3–5 agent archetypes (role, bias, stance range, count)
Output → ScenarioBlueprint
↓
Step 2 — Generate  ·  parallel LLM calls
Agent Generator
  • Pass 1 — one LLM call per archetype, in parallel
  • Each agent gets: name, persona, bias, initial_stance, behavioral_rules[]
  • Pass 2 — one LLM call for the full cast
  • Builds relationship graph: who follows / distrusts / respects whom
  • Diversity check: regenerates if all stances cluster too tightly
Output → list[Agent] with relationships
↓
Step 3 — Simulate  ·  N agents × T ticks LLM calls
Simulation Engine
  • Each tick: all agents reason in parallel (no turn-order bias)
  • Each agent sees: own memory + neighbors' messages + aggregate stance
  • Each agent outputs: stance · action · emotion · reasoning · message
  • AgentMemory stores full history — for_prompt() is the only access point
Output → list[TickRecord] → saved as RunResult JSON
Oracle Loop only — repeats until all coherent or max runs reached
↓
Step 4 — Evaluate  ·  1 LLM call per agent
Evaluator
  • Grades each agent's reasoning coherence
  • Not archetype conformity — coherence: does action match stated reason?
  • No ground truth needed
Output → AgentEvaluation per agent (is_coherent, summary)
failing agents only
↓
Step 5 — Amend  ·  1 LLM call per failing agent
Temple of Learning
  • Adds 1–3 new rules to agent's behavioral_rules[]
  • Additive only — existing rules are never removed
  • Agents accumulate nuance across runs, never regress
Output → Agent with expanded behavioral_rules[]
amended agents re-enter Step 3

Key Decisions

Why a custom engine instead of OASIS / CAMEL-AI?
OASIS is built for social-media simulation. Pythia needs opinion dynamics — different model. Building custom gave full control over prompts, memory, and evaluation with no framework overhead.
Why coherence instead of prediction accuracy?
Prediction accuracy requires ground truth — real-world outcomes to compare against. Coherence (does the agent's reasoning explain its action?) is measurable immediately with one LLM call. No ground truth needed.
Why are behavioral_rules[] additive — never removed?
Removing rules makes agents drift toward generic behavior. Addition accumulates context-sensitive nuance run over run. Agents become richer, not blander.
Why analyze + generate only once in the Oracle Loop?
Agents carry their learned rules forward. Re-analyzing or re-generating would discard everything the amendment step produced. The loop re-uses the same agents across runs.
Why parallel LLM calls per tick?
Sequential calls would introduce turn-order bias — earlier agents influence later ones within the same tick. Parallel calls mean all agents reason from the same world state.
Why an LLMClient Protocol instead of a base class?
Python structural typing — any object with generate(prompt, system) → dict works. No inheritance required. Swap Ollama for Claude by changing one constructor call. The pipeline code never changes.
Why 5 named agents + 290 crowd particles, not 1000 agents?
1000 named agents can't be individually rendered or reasoned about per tick. Protagonists are the characters you follow; the crowd field represents the mass through physics-based particle dynamics.
Why static mock scenarios in the frontend?
Works without Ollama, API keys, or any backend running. Needed for gif recording, first-load UX, and demos on machines with no GPU. The real API overlays on top when available.
